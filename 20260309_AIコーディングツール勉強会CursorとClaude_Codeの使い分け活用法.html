内容整理

### Claude CodeとCursorの比較・違い

青木は、AIコーディングツールとして現在広く使われている「Cursor」と「Claude Code」の違いについて体系的に解説した。

まず課金体系について：Cursorは月200ドルのプランが主流で、月10個程度のツール制作であればプラン上限に達することはないが、それ以上の開発をしたい場合はClaude Codeへの移行が起きる、と青木は説明した。Claude CodeはCursorより「安い」が正確には変動型で、最低30ドル程度から、最大でも大体200ドル程度に収まる上限が設けられている。つまり大量に使う場合はClaude Codeの方がコスト効率が高くなる。

UI・使いやすさの観点では、CursorはGUIのチャット画面で操作でき、スクリーンショットを撮って「これを見て直して」と指示できるため、非エンジニアにも直感的に使いやすい。一方Claude CodeはターミナルベースのCLI（コマンドラインインターフェース）で操作するため、UIが初心者には分かりづらく、エラー対応もスクリーンショットではなくエラーログを読んで対処する必要がある。青木は「エンジニア経験がある人にとっては普通だが、非エンジニアには最初ハードルを感じる」と述べ、個人的にはCursorの方が非エンジニアフレンドリーと評価した。

機能面では「ほぼ一緒」と繰り返し強調されており、どちらを使っても最終的に同等の開発ができる。CursorとClaude Codeは連携も可能で、Cursor上からClaude Codeを起動・操作することが推奨されている。

### CursorからClaude Codeを動かす方法（具体的な操作手順）

青木は実演を交えながら、Cursor上でClaude Codeを動かす具体的な手順を説明した。

1. まずCursorのチャット画面で「このスペースにClaude Codeをインストールしたい」と自然言語で指示する。Cursorが自動でインストール作業を実行する。

2. Cursorのメニューバーにある「ターミナル」ボタン（英語表記の場合は「Terminal」）を押し、「新しいターミナル」を開く。

3. そのターミナル内でClaude Codeが起動し、チャット形式で開発指示が出せるようになる。

4. この状態でClaudeCodeに指示を出すと、課金はCursorではなくClaude Code側に発生する。Cursorのチャット画面で指示を出した場合はCursorに課金が入る。

また、日本語化についても触れており、Cursorのインターフェースが英語で使いにくいと感じる参加者向けに「日本語にするCursorのプラグインを入れて、と言えば日本語になる」と案内した。

### Claude CodeのRun Everything設定・GitHub連携

Claude Codeには「run everything」という設定があり、これを有効にするとライブラリのインストールや各種システム操作を全て自動で実行してくれる。これにより開発者が逐一確認・承認しなくても開発が進む。

また、GitHubとの連携についても説明された。Claude Codeが起動しているターミナル環境でGitHubと連携することで、バージョン管理をしながら開発を進めることができる。さらに、途中までCursorで開発したプロジェクトをClaude Codeで引き継いで続きを開発することも可能。ルールやスキル設定も「CursorのルールをClaude Codeに適用したい」と言うだけで移行できる。

### Claude Code Webとリモート操作・自律開発の潮流

青木は「Claude Code Web」という機能についても紹介した。これはパソコン上でClaude Codeをローカル操作するのではなく、GitHubにコードを置いておくことでスマートフォンやタブレットなどオンラインから遠隔で開発指示を出せる機能。

さらに、この文脈でClaudeCodeを「遠隔操作して自律的に開発させる」トレンドについても言及した。具体的には「オープンクローズ」（Open Clineと思われる）と呼ばれる手法で、LINEの公式アカウントなどと連携しておくと、LINEでメッセージを送るだけでパソコンが反応して自動で開発を開始する、という自律エージェント的な活用が可能になる。青木はこれがX（旧Twitter）で非常に話題になっていると述べ、「ただし、そこまでやりたくない人は無理にやらなくていい。まずはCursorを使いこなすことを優先してほしい」と補足した。話題になっているので「Claude Codeというものを知っておいた方がいい」とは強調した。

### APIコストとモデル選択に関するトラブルシューティング

後半の断片的な会話の中で、APIコストについての言及があった。O3以上のモデルを使うとAPIコストが非常に高くなるという課題が共有された。また、最新モデルに固執するとコストが膨らむ一方、古いモデルに戻すと精度が落ちてうまくいかないケースもある。対策として、GeminiのGemini 2.5モデルへの切り替えがコスト削減の一手として提案された。Speaker Sは「最新モデルを試しながら進めるのがいい」と述べた。

「30万3000円を捨ててもらえる」という発言も断片的に登場しており、開発コストや投資判断に関連する文脈と思われるが、詳細は不明。モデル名については「クロードでいいんじゃないか、精度は高い」という意見も青木から出た。
