内容整理

### AI活用レベルの段階的進化モデル（レベル1〜7）

Speaker A（講師）は冒頭、AIの活用レベルを7段階に体系化して解説した。

**レベル1**は、ChatGPTに分からないことを聞くだけの最も初歩的な使い方。単純な質疑応答の域を出ない段階。

**レベル2**は「ゴールシーク」の概念を活用する段階。ゴールシークとは、現在地とゴールを明示した上で「ここからここまで到達するための道筋を全部考えてくれ」とAIに考えさせるアプローチ。講師はこの概念を分かりやすく実装しているサービス事例として「アドネス（三上君の会社）」を挙げた。アドネスは基本的にゴールシークの概念をそのまま実装したサービスであると評した。

**レベル3・4**は、AIがワークフローを代わりに実行するフェーズ。壁打ちしながら整理したワークフローをAIに実行させる、いわゆるGPTsと呼ばれる段階。この辺りで「Jamespark（ジェームスパーク）」や「Manus」などのツールが登場してきたと説明。資料の仕上げなど一定の作業をAIに丸投げできるようになる。

**レベル5**は、特化型AIエージェントの登場。Jamesparkのような汎用エージェントに任せられるようになったことで、デザイン専門・営業支援専門など特定用途に特化したAIエージェントが出現してきた段階。

**レベル6・7**が今回のハンズオンの核心。システム開発までAIが全部やれるようになった段階。これによって従来超えられなかった壁、すなわち「外部サイトを使わないと完遂できないワークフロー」を全て自動化できるようになる。ChatGPTやGeminiだけでは、リサーチは自分でやってからAIに渡す、という分断があったが、システム開発ができることでこの分断がなくなる。

講師は自分のYouTube業務を例に具体的なユースケースを提示した。YouTube上でのリサーチ→バズ分析→企画・台本生成という一連の流れを、講師自身の思考プロセスをAIに教育した上で全て自動化できると説明。ほぼ全ての作業を仕組み化できる世界線に近づいてきていると述べた。

講師がこのシステム開発手法を習得したのは11月27日（講座の約3ヶ月弱前）。習得当初は1システム開発に5〜6時間かかっていたが、現在は2時間以内に短縮されている。AIの進化スピードが極めて速いことの証左として紹介された。さらに、このイベント直前にOpusモデルが4.5から4.6にアップデートされたことで、当日はさらに簡単になっているという「僥倖」があったと講師は述べた。

### Skillsという新概念とポケモン的AIの世界観

レベル6・7の話に加え、講師は最近登場した「Skills（スキルズ）」という新概念を紹介した。スキルズは1つ1つがGPTsに近い概念で、特定のワークフロー作業を「スキル（必殺技）」として登録できる仕組み。

講師はこれをポケモンに例えた。「ポケモンに必殺技を覚えさせるように、AIにスキルをどんどん登録できる。登録したスキルをAI自身が状況を判断して自律的に選択・実行する」という世界線に来ていると説明。この1ヶ月でAIを「シミュレーションゲーム」のように感じているとも述べた。

さらにシステム開発とスキルズを組み合わせることで、ほぼ全ての仕事をAIに任せられる世界線にさらに近づいていると述べた。自律型の「ペット型AI」という概念にまで話が及んだが、時間の都合で後半での説明に回すとした。

今日の目標は「レベル6・7まで体験してもらうこと」「2つのシステムを作って本番環境までデプロイして帰ること」と明示された。

### システム開発の肝：APIのつなぎ込みとその意義

システム開発における最重要ポイントとして「API連携」が強調された。外部サイトを使わずに完結する作業ならば従来通りGeminiやChatGPTで十分だが、外部サイトを経由しないと完遂できないワークフローがある場合にこそシステム開発が本領を発揮する。

講師は「APIというものは拾い放題である」「何でもある」という重要な認識を参加者に伝えた。公式APIと非公式APIの両方が存在することにも言及し、例えばインスタグラムの情報を取得したい場合、公式APIにバカ真面目にこだわる必要はなく、外部のAPIサービスを組み合わせる発想が重要だと伝えた（ただし公式APIでは情報取得に制限がある点は確認事項として提示された）。

APIキーの知識量がシステム開発の質を決定するという点も強調。「APIキーの知識をどれだけ知っているかがキーになる」と明言した。

### 開発環境のセットアップ：CursorとGitHub

ハンズオンの実施にあたり、参加者はまず以下2つのセットアップを行った。

1. **Cursorのインストール**: Cursorは真っ黒の画面（ターミナル）が出る開発環境ツール。「慣れると好きになる」と講師が述べた。WindowsユーザーとMacユーザーが混在しており、Windowsが1〜2名いることが確認された。

2. **GitHubアカウントの作成**: GitHubはシステム開発で生成される大量のプログラミングコードをオンライン上に保存する場所。パソコンが吹っ飛んでもデータが残ること、社内メンバーへの共有が容易になることがメリットとして挙げられた。「リポジトリ」はGitHub上の「フォルダ」にあたる概念として説明。公開・プライベートの2モードがあり、プライベート設定にしないとコードが全世界に公開されてしまうため、プライベートモードを推奨した。

完了した参加者はオープンチャットにコメントするよう指示があり、資料リンクもオープンチャット最上部に掲示されている。困っている参加者は声をかけるよう促された。

### システム開発の実演：プランモードと要件定義

Cursorを開いた後の実際の開発フローを講師がデモ形式で実演した。

**手順の概要:**

1. Open Projectでフォルダを選択・新規フォルダ作成

2. GitHubをつないで新しいリポジトリを作成し、プライベートモードで起動

3. Cursorの画面でモード選択（エージェント・プラン・デバッグなど）

4. まず「プランモード」で要件定義から開始

・モデルはOpus 4.6を推奨（オートモードを解除して指定）

・「魔法のランプ」のように、作りたいものを好き放題に自然言語で伝える

**デモで使われた要件定義の具体例:**

「インスタグラムで競合アカウントのURLを貼り付けたら、直近1年間のリール動画をリサーチし、そのアカウントの直近1年間の投稿の中で平均再生数の2倍以上あるバズ投稿のみ抽出し、その投稿を全自動で文字起こしし、文字起こし結果を台本の型テンプレートとして生成するところまで、全部自動で一気通貫でやりたい」

プランモードでこれを入力すると、AIが自動的に実現方法を考えて提示してくれる。この段階でAIから「インスタの公式APIでは情報取得に制限があるため外部サービスが必要」などの確認事項が返ってくることも実例として示された。

**エンジニアと違う点の強調:**

・「パワハラし放題」であること：無茶な要件をぶつけても怒らない

・仕様変更が容易：走り出した後にやっぱりこうしたいと言っても対応してくれる

・途中で止めて技術要件を追加することも可能（Speaker Kが補足：「ごめん、技術要件これでやっておいて」と途中で足すだけで大丈夫）

・機能の後付け追加も簡単

### 技術要件の「魔法の言葉」

Speaker A・Speaker K両者が強調した重要な技術要件として、要件定義時に必ず含めるべき「魔法の言葉」が紹介された。

・**「Next.js」**：Webアプリ開発のフレームワーク

・**「本番環境はGoogle Cloudを想定している（GCP）」**：本番デプロイ先の指定

Speaker Kは「デザイン専門のAIもあるし、惚れ惚れするような魔法の言葉」と表現し、要件定義時にこれらを追加しておくと後の工程が大幅に楽になると強調した。最低でもこの2つだけ指定しておけば十分とのこと。オープンチャットにテンプレートのリンクも掲示されており、コピー＆ペーストして使うよう案内された。

### 文字起こし機能：Groq Whisperの推奨

参加者の多くが文字起こし機能を実装しようとしていたことを受け、Speaker Aが重要な技術情報を提供した。

AIが通常提案してくる「OpenAI Whisper」よりも「Groq Whisper（Faster Whisper）」の方が性能・速度ともに優れており、文字起こしスピードが約5倍速いと説明。APIキーの取得方法もCursorに「Groq Whisperを使いたいからAPIキーの取り方を全部教えて」と聞けば全部教えてくれるため、文字起こし機能を実装する際はGroq Whisperを使うことを強く推奨した。

### 本番環境へのデプロイ（GCP）

開発が完了したシステムを本番環境（Google Cloud Platform）にデプロイする手順も説明された。

1. システムを動かしてエラーを見つけ、スクショを撮って修正→修正→修正をやり切る

2. 「本番環境のGCPにデプロイしたい」と入力すると、本番で使えるURLが発行される

3. 初回はGoogleクラウドサーバーへのアップ時に課金情報の入力が必要になる場面がある

4. やり方が分からなければ「できるところまで全部やって」と言うとGoogle Cloud CLIを使って自動対応してくれる

5. 課金情報の入力など最終的な手作業は自分で行う必要がある

講師は「とにかく作業したら負けという気持ちで、なるべくAIにやらせる姿勢を貫け」と強調。ここまで完了すると「俺デプロイしたよ」という状態、つまりエンジニアと同等のアウトプットが出せる状態になると述べた。
